Phase 1 Part 2

Use Case 1: Player Moves Main Character
Primary Actor: Player (Robber)
Goal in context: Navigate the main character through the maze to collect rewards and reach the exit.
Preconditions: The game has started, and the player controls the robber on the board. The player score starts at 100.
Trigger: The player presses a movement key (WASD) on the keyboard.
Scenario:
1.	Player presses one of the movement keys (WASD).
2.	The system checks if the move is valid (no barriers or walls in the way).
3.	If valid, the main character moves one cell in the intended direction.
4.	If the character lands on a reward, the system updates the score and removes the reward from the map.
5.	The game displays the updated score and playtime.
Exceptions:
•	If the move is invalid (blocked by a wall or barrier), the main character stays in place.
•	If the main character lands on a punishment, the player’s score is penalized, and the punishment is removed. If the score drops below zero, the game ends.
Priority: High
When available: After game start
Frequency of use: Constant (throughout the game)
Channel to actor: Keyboard input
Secondary actors: None
Channels to secondary actors: N/A
Open issues: None

Use Case 2: Main Character Collects Reward
Primary Actor: Player (Robber)
Goal in context: Collect all the basic rewards (money) to proceed to the exit and win the game.
Preconditions: The main character must move to a cell containing a reward.
Trigger: The player moves the robber to a reward-containing cell.
Scenario:
1.	The player moves the robber to a cell with a money reward.
2.	The system detects the reward, updates the player’s score, and removes the reward from the cell.
3.	The game updates the score on the screen.
Exceptions:
•	If the reward is collected but all rewards have not been gathered, the player cannot proceed to the exit.
Priority: High
When available: Throughout the game
Frequency of use: Multiple times (once per reward)
Channel to actor: Visual feedback on the screen (score update)
Secondary actors: None
Channels to secondary actors: N/A
Open issues: None

Use Case 3: Main Character Avoids Enemy
Primary Actor: Player (Robber)
Goal in context: Avoid getting caught by moving enemies (guards and dogs) to stay in the game.
Preconditions: The game is in progress, and enemies are actively moving on the board.
Trigger: The enemy moves toward the robber's location or the player moves the robber toward an enemy's path.
Scenario:
1.	The player moves the robber to a safe cell, avoiding enemies.
2.	The enemy (guard or dog) moves according to its logic, either patrolling or chasing the robber.
3.	The game checks for proximity between the robber and enemies.
4.	If the robber is caught by an enemy, the game ends.
Exceptions:
•	If the player gets caught by an enemy, the game ends immediately.
Priority: High
When available: Throughout the game
Frequency of use: Constant
Channel to actor: Visual feedback (enemy proximity)
Secondary actors: Enemies (guards, dogs)
Channels to secondary actors: Enemy movement logic
Open issues: Tuning the enemy movement patterns to balance difficulty

Use Case 4: Timer Management
Primary Actor: System
Goal in context: Track the game time and limit the player’s available time to complete objectives.
Preconditions: The game is in progress, and the timer has started.
Trigger: The game starts.
Scenario:
1.	The system starts a timer when the game begins.
2.	The game constantly updates and displays the remaining time on the screen.
3.	If the timer reaches zero before the player reaches the exit, the game ends, and the player loses.
Exceptions:
•	If the player collects all basic rewards and exits the vault before time expires, the player wins.
Priority: High
When available: Throughout the game
Frequency of use: Continuous
Channel to actor: Visual feedback (time display on the screen)
Secondary actors: None
Channels to secondary actors: N/A
Open issues: Determining appropriate time limits for difficulty balancing
Use Case 5: Triggering the Exit
Primary Actor: Player (Robber)
Goal in context: Reach the exit after collecting all basic rewards (money).
Preconditions: All basic rewards (money) must be collected, and the player is at the exit cell.
Trigger: The player moves the robber to the exit cell after collecting all money.
Scenario:
1.	The player collects all the money rewards on the map.
2.	The exit becomes accessible to the player.
3.	The player navigates the robber to the exit cell.
4.	The game checks if all rewards are collected, and if true, the player wins.
Exceptions:
•	If the player reaches the exit without collecting all basic rewards, the exit remains locked.
Priority: High
When available: Endgame, after collecting all rewards
Frequency of use: Once per game (if successful)
Channel to actor: Visual feedback (exit unlocked)
Secondary actors: None
Channels to secondary actors: N/A
Open issues: Defining the visual indication that the exit is available
